IP Scoring Pipeline Overview

The Targeting Infrastructure team is responsible for assigning scores to IP addresses based on their site visit behavior. These scores determine which IPs can be targeted in advertising campaigns and at what priority level. This document explains how site visits flow through our system and how scores are assigned.

Data Flow

The scoring pipeline processes data through the following stages:

  1. Site Visit Signal — IP addresses and their site visits are collected from GUID logs and direct data providers. Runs hourly (some vendors have different cadences).
  2. Categorization (IPDSC) — Each site visit is categorized via tpa_ipdsc_export, starting ~2:45 AM UTC daily. Key sources: DS13 (bucket/vertical) and DS19 (keywords).
  3. Scoring — Advertiser and Campaign scores calculated based on bucket/vertical/keyword matches. Waits for DS13/DS19, runs ~3:30 AM UTC. Scores sent directly to BIDDER (not through AT). Uses 30-day lookback.
  4. MembershipDB — IP/DS/DSC combos from Categorization job loaded ~3:45 AM UTC (last step of tpa_ipdsc_export). This is batch association of categories to IPs; real-time association also happens for real-time events. Outputs a refresh of segments every 4 hours on the 4hr mark UTC (00:00, 04:00, 08:00, 12:00, 16:00, 20:00).
  5. Bidder (Aerospike) — Scores are written to Avro files in GCS, with messages published to RabbitMQ and Kafka (by MembershipDB), pointing at files to read or with explicit updates to process. BIDDER's membership consumer loads those updates into Aerospike (fancy key-value cache, like Redis but cheaper for large volume) for real-time bidding decisions. Scores loaded after TI's audience_intent job runs, then BIDDER runs a job to populate Aerospike. (POC: Forrest) Bidder uses this data for a received auction's IP to lookup scores via waterfall: RTC, or campaign, or fallback to advertiser, then compares that score against threshold to determine whether or not to bid.

Three Independent Pipelines

The system consists of three independent pipelines that BIDDER references separately. An IP can be in a segment and biddable WITHOUT having a score. TI only handles Prospecting campaigns; retargeting data (e.g., guid_log for advertiser page views) also feeds into MembershipDB but is outside TI's scope.

BIDDER references both segment membership and Scores independently. These pipelines are not dependent on each other, which means an IP can enter a segment before it receives a score.

MembershipDB Dependencies: IPDSC, vast_start/vast_impression topics, guidv2/conversion/clickpassv2 topics, bidder-beeswax-win-notifications topic, geo data. If an IP comes in through any of these, an audience expression can run against it. See: github.com/SteelHouse/membership-db

Score Types and Paths

  • Advertiser scores — Calculated by TI, sent directly to BIDDER (not through AT).
  • Campaign scores — Calculated by TI, sent directly to BIDDER (not through AT). Takes precedence over advertiser scores.
  • RTC scores (Real-Time Conquesting) (Segment Score) — Come from MembershipDB, sent by AT in real-time to BIDDER. Populated during real-time events. TTL: 24–28 hours depending on timing.

URL Categorization Process

When a site visit arrives (e.g., http://nike.com/shoes with IP 192.123.123.12), the system follows this process to assign a vertical:

  1. Extract domain — The domain (nike.com) is extracted from the full URL.
  2. Check blocklist — If the domain is on the ecommerce blocklist, processing stops.
  3. Check whitelist
  4. Look up domain in verticals

Data Hierarchy

Understanding how IPs connect to campaigns:

  1. Data Sources (IPDSC) — IPs are mapped to category_ids within each data source. IPDSC is the umbrella term for all data sources, stored as GCS buckets separated by date.
  2. Category IDs — Each data source has its own category_ids. These are essentially audiences within that data source. Queryable in tpa.categories.
  3. Audiences — Created by combining category_ids from various data sources (both in-house and third-party).
  4. Campaigns — Built by including or excluding specific audiences.
  5. Advertisers — Own campaigns and are grouped under verticals.
  6. Verticals — Industry sub-categories that advertisers belong to.

Scoring Logic

Scores are assigned based on three factors: whether the IP falls within the target bucket (industry), whether it falls within the target vertical (sub-industry), and whether it matches campaign keywords. The combination determines the score tier. The scores below reflect Campaign level scores (prospecting_intent); Advertiser level scores are slightly different.

Score Tier Score Value Criteria
High Intent 10,000 IP matches bucket, vertical, AND keywords. Highest priority.
Peak Performance 8,000 IP matches bucket and vertical, but not keywords.
Mid Intent 3,333–6,665 IP matches bucket only. Ranked by page views and recency within the bucket.
Max Reach -1 IP matches keywords only (not in bucket or vertical). Random score in range.
Note: Max reach scoring has been turned off. IPs that would have been scored 1–3332 (keyword match only) now remain at -1. This was intentional — unscored IPs outperformed scored max reach IPs, and the job cost ~$150–200/day in compute.

Lookback windows: Both max reach and high intent use 30-day lookback via IPDSC.

Advertiser vs. Campaign Scores

Every IP receives an advertiser-level score. On campaign creation the campaign-level score will be created once enough information has been gathered. It is then preferred over the advertiser score for bidding purposes. Both scores are stored, but the campaign score takes precedence. This allows for more granular targeting at the campaign level while maintaining a baseline at the advertiser level.

Scores vs Verticals: These are conceptually separate. An IP can have an RTC score without matching a vertical. RTC updates can also add verticals. Current campaign scoring happens to be based on verticals, but they're independent concepts.

Why Unscored IPs Are Expected

Unscored IPs (-1 in bid_price_log) are a normal part of the system. Key insight: Analysis showed unscored IPs actually outperform scored max reach IPs, which is why max reach scoring was disabled. Here are the expected reasons an IP may be unscored:

  • Keyword-only match (most common) — IP matches keywords (DS19) but NOT bucket/vertical (DS13). MembershipDB includes them in the segment, but Scoring has nothing to score. With max reach scoring disabled, these remain at -1. This is expected behavior. For campaigns targeting MM KW (keywords only), ~50% unscored is normal.
  • Timing gap — IP visits after the daily scoring job (~3:30 AM UTC) but before the next run. MembershipDB may pick it up in its 4-hour cycle, allowing bidding before scores exist. IPs that visit after the daily scoring job typically get caught by RTC and receive a conquest_score in real-time, rather than remaining unscored until the next batch run.
  • Real-Time Conquesting (RTC) — RTC can assign IPs to verticals in real-time with a conquest_score, but they won't get a batch score until the next daily run.
  • Market bids / new IPs — Some IPs come from market bids that we've never seen in our ecosystem. The augmentor log only processes the last 7 days.
  • Domain not in verticals lookup — Domain passes blocklist/whitelist/model checks but isn't in website_crawl_verticals. The IP never gets a vertical assignment. (Known gap / area for improvement.)

Audience Expression and Thresholds

Bidder abstraction: Bidder has a list of IPs associated to a campaign (based on audience expression). Those IPs can have scores or not. Some IPs in the expression may be in the vertical, some are not — depends on what the expression is defined as.

A campaign's audience can include 3rd party IPs + high intent IPs (if enabled). Bidder/Perf team adjusts the score threshold based on performance and pacing. When threshold drops to max reach (-1), ALL IPs in the audience can be bid on regardless of score.

Score Assignment Logic

Once categorized, the IP is scored based on these checks:

Check Result Impact
In Bucket (industry)? ✅ Qualifies for Mid Intent (3,333–6,665)
In Vertical (sub-industry)? ✅ Qualifies for Peak Performance (8,000)
Matches Keywords? ✅ Qualifies for High Intent (10,000)

Final Score: 10,000 (High Intent) — This IP matches bucket, vertical, AND keywords.

At Bid Time

Score priority (WATERFALL, not additive):

  1. RTC score (highest priority)
  2. Campaign score
  3. Advertiser score (fallback)

Bidder picks the highest priority available. Scores are NOT combined. The selected score is compared to the campaign threshold. If the score meets or exceeds the threshold, the bid proceeds.

Score routing:

  • Advertiser & Campaign scores → sent directly to BIDDER by TI (not through AT)
  • RTC scores → sent by AT in real-time to BIDDER

Key Concept: Three Independent Pipelines

Understanding why unscored IPs exist requires understanding that BIDDER references three independent pipelines:

Pipeline What It Does Cadence
IPDSC Categorizes IPs into DS13 (verticals), DS19 (keywords) Daily ~2:45 AM UTC
Scoring Assigns scores based on bucket/vertical/keyword matches Daily ~3:30 AM UTC
MembershipDB Evaluates audience expressions, determines segment membership Every 4 hours

These pipelines are NOT dependent on each other. An IP can be in a segment (MembershipDB) and biddable WITHOUT having a score (Scoring). This is by design.

Bidder abstraction: Bidder has a list of IPs associated to a campaign (based on audience expression). Those IPs can have scores or not. Whether an IP is in a vertical depends on the expression definition.

Where Unscored IPs Happen

An IP can appear as unscored (-1) in bid_price_log in the following scenarios:

Scenario 0: Keyword-only match (MOST COMMON)

This is the primary reason for unscored IPs, especially in campaigns targeting MM KW (keywords only):

  • IP matches keywords (DS19) but NOT bucket/vertical (DS13)
  • MembershipDB includes the IP in the segment
  • Scoring has nothing to score → remains -1
  • With max reach scoring disabled, these keyword-only matches stay at -1
This is expected behavior, not a bug. For campaigns like CID 71353 (MM KW only), ~50% unscored is normal.

Scenario 1: Visit occurs after daily jobs run

If an IP visits a site after the daily scoring jobs complete, it typically falls into the RTC bucket and gets a conquest_score in real-time. This means the timing gap is usually NOT a cause of unscored IPs. RTC handles this in real-time, so an IP visiting at 2pm will typically get a conquest_score (TTL: 24–28 hours) before the next batch run.

However, if RTC doesn't catch it (e.g., URL not classified), the timeline would be:

Time (UTC) Event
~2:45 AM IPDSC runs (tpa_ipdsc_export)
~3:30 AM Scoring job runs
~3:45 AM MembershipDB loaded
2:00 PM IP visits site (missed today's batch)
3:00 PM IP captured by site_visit_signal
Next ~3:45 AM IP categorized and loaded to MembershipDB; batch scores calculated

Without RTC: ~13–14 hours where IP would be unscored. But RTC typically catches this in real-time.

Scenario 2: Domain not in verticals lookup

If a domain passes the blocklist/whitelist/model checks but is NOT found in the website_crawl_verticals lookup, nothing happens — the IP never gets assigned a vertical and therefore never gets scored. This is a known area for improvement.

Scenario 3: Domain blocked or fails ecommerce model

If a domain is on the blocklist, or if it's not whitelisted and scores below 0.4 on Tucker's ecommerce model, processing stops and the IP is never categorized.

Scenario 4: Real-Time Conquesting (RTC)

RTC can assign IPs to verticals in real-time with a conquest_score, but they won't get our batch score until the next daily run. Check the conquest_score and conquest_score_ttl columns in bid_price_log to identify these.

Scenario 5: Market bids / new IPs

Some IPs come from market bids that we've never seen in our ecosystem. The augmentor log only processes the last 7 days, so brand new IPs may be bid on before they're categorized.
