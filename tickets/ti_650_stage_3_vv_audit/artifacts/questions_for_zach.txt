4. Data retention
-----------------

The row-level table will grow by millions of rows per day (all
advertisers × all VVs). Three options:

OPTION A: 90-day rolling window
  - Delete partitions older than 90 days each daily run.
  - Matches cost_impression_log's 90-day TTL pattern.
  - Pro: table stays manageable, consistent with existing conventions.
  - Con: lose row-level drill-down for older data.

OPTION B: Keep everything
  - Never delete. Storage grows linearly.
  - BQ handles it fine with partitioning + clustering.
  - Pro: enables year-over-year trend analysis at row level.
  - Con: ever-growing storage cost, unclear if row-level granularity
    is needed beyond 90 days.

OPTION C: Hybrid — roll row-level, keep summaries
  - Row-level table = 90-day rolling window (Option A).
  - Daily summary views/tables = permanent (mutation rate,
    ft_null rate, cross-device % by advertiser/campaign/day).
  - Pro: best of both — cheap long-term storage with drill-down
    for recent data.
  - Con: slightly more infrastructure (summary table + daily agg job).

My recommendation: Option C. Row-level data older than 90 days
is rarely needed for ad-hoc investigation. Summary aggregates are
tiny and give long-term trend visibility.

Questions:
  a. Which option?
  b. If Option C: what dimensions should the summary table aggregate
     by? (advertiser, campaign, day) seems minimal. Add device_type?
     cross_device? NTB flags?
  c. If we go with a rolling window: do we need a one-time "archive"
     snapshot before deleting old partitions, or is summary sufficient?

8. first_touch_ad_served_id — OR vs AND
----------------------------------------

Sharad confirmed (2026-03-04) the first_touch lookup searches on
"both bid_ip AND ip of the attributable impression."

Open question: does "both" mean:
  - OR: either bid_ip or impression ip matching a Stage 1 record
    = found (more resilient to partial mutation)
  - AND: both must match (more restrictive, any mutation breaks it)

This matters for quantifying mutation's contribution to the 40%
NULL rate.

Our A9a results show:
  - ft_null when mutated:     54.85%
  - ft_null when NOT mutated: 38.19%  (+16.66pp delta)
  - But 74,452 of 87,917 ft_null VVs had NO mutation

So mutation is a contributing factor (~15% of NULLs) but not the
primary driver. Understanding OR vs AND would explain the mechanism.
