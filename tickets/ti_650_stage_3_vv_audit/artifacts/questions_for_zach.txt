8. first_touch_ad_served_id — OR vs AND
----------------------------------------

Sharad confirmed (2026-03-04) the first_touch lookup searches on
"both bid_ip AND ip of the attributable impression."

Open question: does "both" mean:
  - OR: either bid_ip or impression ip matching a Stage 1 record
    = found (more resilient to partial mutation)
  - AND: both must match (more restrictive, any mutation breaks it)

This matters for quantifying mutation's contribution to the 40%
NULL rate.

Our A9a results show:
  - ft_null when mutated:     54.85%
  - ft_null when NOT mutated: 38.19%  (+16.66pp delta)
  - But 74,452 of 87,917 ft_null VVs had NO mutation

So mutation is a contributing factor (~15% of NULLs) but not the
primary driver. Understanding OR vs AND would explain the mechanism.


9. Which IP populates each stage's segment?
---------------------------------------------

Our data shows VAST playback IP (not bid IP) feeds into the
next-stage targeting segment. Tiebreaker test on 7,546 VVs
where ft_bid_ip != ft_vast_ip:
  - 70.5% of subsequent bids came from the VAST IP
  - 23.8% came from the bid IP
  - 5.7% matched neither

This confirms the MES diagram (green line = VAST IP → next stage).

Questions:

  b. Stage 3: after a VV, which IP gets added to Stage 3 segment?
     The redirect IP (clickpass.ip)? The visit IP (ui_visits.ip)?
     Or the VAST IP of the attributed impression?
  c. Does this vary by inventory type (CTV vs non-CTV)?

RESOLVED (meeting_zach_3, 2026-03-04)
================================================================

- Retention: 90-day rolling window confirmed.
- Pipeline: SQLMesh job confirmed.
- Table location: ask D Platt.
- Intermediate impressions: "Do we care about those? No."
  Only first-touch and last-touch matter.
- Campaign ID = Stage: one-to-one. Use campaigns.funnel_id or
  campaign_template_id. Objective ID: 1=S1, 5=S2, 6=S3.
- IPs accumulate stages: never removed from prior stages.
  Frequency capping (14-day) handles dedup, not targeting removal.
- Budget split: S1 ~75-80%, S2 ~5-10%, S3 = remainder (highest intent).
- Bidder has no concept of stages: treats each campaign independently.
- VV attribution = stack model: impressions on stack, page view
  checks top of stack (most recent), everything behind is ineligible.
- Non-CTV: use impression_log instead of event_log for display.
- Table supports all stages: not just Stage 3. Stage 1 VV = Stage 2/3
  columns NULL. Stage 3 VV = entire row full.
